## 01 从Claude Code的Grep说起
*   **核心观点**：Claude Code创建者公开表示，该工具不使用RAG、Embedding或向量数据库，而是依赖LLM驱动的Grep（及Glob）进行代码搜索，效果优于传统RAG方案。
*   **决策背景**：
    *   早期版本曾使用RAG+本地向量库，但发现Agent式搜索效果更好。
    *   决策部分基于直觉，也有在Meta观察到的工程师行为（IDE功能崩溃后回退手动Grep）的启发。
*   **本文目标**：基于Claude Code泄露源码，从实现层面拆解其零索引内容搜索机制、设计哲学及行业对比。

## 02 LLM驱动的多轮循环搜索机制
### 2.1 核心循环与工具集
*   **机制**：一个由LLM自主决策的搜索循环：用户输入 → LLM判断（返回文本或工具调用）→ 执行工具 → 结果追加到上下文 → 再次交予LLM判断，直至信息充足。
*   **四大核心工具**：
    *   **GrepTool**：基于ripgrep (`rg`) 的正则内容搜索。
    *   **GlobTool**：基于glob模式匹配的文件名/路径查找。
    *   **FileReadTool**：读取指定文件的指定行范围。
    *   **AgentTool**：启动独立子Agent（如只读的`Explore`类型）进行多步探索，实现上下文隔离，主对话仅接收结论。

### 2.2 GrepTool的信息量控制
*   **设计哲学**：用软引导代替硬约束，让LLM自主决定搜索路径。
*   **三种输出模式**：
    *   `files_with_matches`（默认）：仅返回匹配文件路径，强制LLM后续用`Read`读取内容以控制上下文消耗。
    *   `content`：直接返回匹配行的上下文代码。
    *   `count`：返回每个文件匹配数量，用于评估分布密度。
*   **保护机制**：`head_limit`参数防止海量结果淹没上下文窗口。

### 2.3 实战：追踪工具执行记录
*   **案例**：展示了一个典型的四轮搜索过程（Grep定位 → Read查证 → 再Grep定位 → 最终Read），体现了LLM主动、迭代的搜索模式。

## 03 为什么暴力搜索能奏效：ripgrep的速度秘密
*   **底层实现**：`GrepTool`底层调用Rust编写的高性能搜索工具`ripgrep`。
*   **五层过滤机制**（乘法叠加，极速缩小搜索范围）：
    1.  **目录级剪枝**：遵循`.gitignore`规则。
    2.  **路径范围限制**：通过`path`参数限定搜索起点。
    3.  **文件类型过滤**：通过`glob`参数跳过不匹配文件。
    4.  **二进制文件检测**：快速跳过非文本文件。
    5.  **正则匹配内容搜索**：最后执行实际匹配。
*   **文件内搜索加速**：
    *   **SIMD向量化匹配**、**Boyer-Moore算法**、**操作系统页缓存**、**`mmap`零拷贝**、**多线程并行**。
*   **性能实测**：ripgrep速度远优于GNU grep。
*   **数据规模差异**：本地代码库规模远小于网络搜索，暴力扫描完全可行。

## 04 Grep方案的真实成本与收益
### 4.1 与Cursor的对比
*   **Claude Code (零索引/纯Grep)**：LLM实时驱动文本搜索。
*   **Cursor (双索引/混合搜索)**：维护稀疏和密集索引，但内部也将`grep_search`标记为“主要探索工具”。
*   **结论**：精确匹配找已知符号远高频于语义理解找相似概念。

### 4.2 Codex的验证
*   **共识**：OpenAI的Codex CLI同样选择零索引、不用向量数据库。
*   **分歧**：实现路径不同。Claude Code封装专用工具，Codex让LLM自由组合shell命令。

### 4.3 Grep方案的成本问题与应对
*   **成本（Token消耗）**：多轮循环会将大量代码载入上下文。
*   **优化/应对**：
    *   **并行调用**，**子Agent隔离**中间过程。
    *   **Prompt复用**：通过Prompt缓存可实现高达92%的输入复用率。
    *   **LSP引入**：用“跳转定义”等语义操作替代部分搜索，降低约40%的Token消耗。

### 4.4 Grep与语义搜索的边界
*   **Grep最适用场景**：代码库中程序员留下的函数名、类名等精确锚点。
*   **技术互补**：在纯自然语言问答中，语义理解仍有优势。LLM查询扩展可提升Grep在此类查询上的表现。

## 05 总结：RAG已死？
*   **对“RAG已死”的回应**：不是广义的RAG范式死亡，而是“预先建索引、静态一次性检索”这种特定实现方式，在代码搜索场景下正被替代。
*   **替代方案的三大支柱**：
    1.  **代码本身是Grep友好的**。
    2.  **本地项目规模撑得住暴力搜索**。
    3.  **Agent带来主动、迭代的检索模式**。
*   **最终定位**：技术选择由数据特性和规模决定，不应该是信仰问题。
