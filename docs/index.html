AgentForge 的核心包 agentcore 遵循一个简单而坚定的原则：只实现最核心的东西，其他一切通过扩展实现。

设计哲学：最小内核，最大灵活性

与 pi-mono 的殊途同归

pi-mono agentcore
核心代码量 ~400 行 TypeScript ~120 行 Python
扩展方式 npm 包组合 pip 包组合
内置工具 零 零
LLM 提供商 按需安装 按需注入

设计模式

  • Protocol（协议）而非继承 — 用户实现自己的 Provider，不依赖任何基类
  • 自注册模式 — 工具在运行时注册，注册者完全掌控自己的代码生命周期
  • 依赖注入 — Agent 本身不创建依赖，外部注入，每个组件都可以独立测试和替换
  • 零强制依赖 — 核心包不依赖任何外部库

为什么重复造轮子是合理的？

  • 学习角度 — 只有亲手实现一遍，才能真正理解 Agent 的核心机制
  • 面试角度 — 120 行代码，可以向面试官逐行讲解
  • 实用角度 — 简单任务用简单工具，不是每个任务都需要全套框架

Agent 的智能来自模型，工程代码的职责是构建 harness。

放到 AgentForge 里：agentcore 负责最小 agent runtime，tools/skills/memory/mcps 负责把模型放进一个可工作的环境。

可靠的 Tool Calling：三层防护

很多人在 prompt 里写"不许乱编参数"，测试几个例子发现没问题，一上线就崩。LLM 的输出本质是概率采样，prompt 只是"软推"而非"硬锁"。

三层解决方案

第一层：优化软约束（Prompt）

提示词应该像一份操作合同，清楚标注每个参数的类型、边界、非法值举例。同时加入 Few-shot 示例，直接给模型看"正确的用户输入 → 正确的工具调用"对照组。

第二层：硬约束（JSON Schema）

不再用自然语言描述参数，而是用机器能验证的结构来定义。关键：additionalProperties: false 至关重要——没有它，模型可以附带一个它"觉得有用"的额外字段。

第三层：兜底机制（校验-清洗-重试闭环）

即便有 Schema，极端情况仍可能发生：模型输出了语法合规但语义荒谬的参数。

架构层面的清醒认识

LLM 只应当承担"决策"职能，而不应当承担"执行"职能。模型负责决策，框架负责让决策可观察、可执行、可约束、可扩展。

Prompt 缓存：从 5% 到 90%+

缓存的到底是什么？

缓存的不是输出文本，而是 Transformer 注意力层里的 Key/Value 张量。prompt caching 缓存的就是 prefill 阶段算出来的 K/V，跳过 prefill 省算力 + 降首 token 延迟。

三家 API 开法对比

平台 启用方式 起步门槛 TTL 策略
Claude 显式 cache_control 1024 token 5min（默认）/ 1h
OpenAI 自动 1024 token 5-10min（闲时 1h）
DeepSeek 自动 64 token 几小时到几天

命中率只有 5% 的六大元凶

  1. 时间戳/动态变量混进系统区 — session ID、user_id 永远不要放在系统区
  2. JSON/工具定义不稳定序列化 — 序列化时强制 sort_keys=True
  3. Cache breakpoint 打在变化的 block 上 — cache_control 标在最后一个不变 block
  4. 模式切换换工具集 — 用 EnterXxxMode/ExitXxxMode 工具代替换工具集
  5. 往前缀里追加 reminder — 用 <system-reminder> 标签包起来塞到下一条 user message
  6. 模型版本悄悄变了 — 缓存是模型粒度的，换版本缓存全部失效

90%+ 的七条工程纪律

  1. 把 prompt 拆成「不变区」和「变量区」，cache_control 标在不变区末尾
  2. 所有动态信息走「追加」不走「修改」
  3. 工具集稳定，工具序列化稳定
  4. 长会话每隔十几个 block 打中间断点
  5. 多 Agent 共享前缀模板
  6. 监控按 stream 切分，不看聚合数
  7. 把缓存破裂当事故响应

意图识别技术全景

意图识别，本质上是一个文本分类问题，是对话系统的"方向盘"。核心挑战在于如何判断当前查询与历史对话的上下文关联。

单轮 vs 多轮方法

场景 推荐方法
单轮 规则匹配 → 向量检索 → BERT微调 → LLM (Prompt/SFT/Function Calling)
多轮 Pipeline（上下文语义分割/槽位关联）→ 端到端大模型

融合方案是工业级落地最佳实践

规则(高优先级) → 小模型(BERT) → 向量检索/大模型(兜底) — 兼顾速度、成本、准确率和覆盖率。

RAG vs Grep：代码搜索的新范式

Claude Code 创建者公开表示，该工具不使用 RAG、Embedding 或向量数据库，而是依赖 LLM 驱动的 Grep 进行代码搜索，效果优于传统 RAG 方案。

LLM 驱动的多轮循环搜索

一个由 LLM 自主决策的搜索循环：用户输入 → LLM 判断（返回文本或工具调用）→ 执行工具 → 结果追加到上下文 → 再次交予 LLM 判断，直至信息充足。

为什么暴力搜索能奏效？

  • ripgrep 的速度秘密 — SIMD向量化、Boyer-Moore算法、操作系统页缓存、mmap零拷贝
  • 本地项目规模撑得住暴力搜索 — 本地代码库规模远小于网络搜索
  • 代码本身是 Grep 友好的 — 函数名、类名等精确锚点

Grep 与语义搜索的边界

Grep 最适用场景：代码库中程序员留下的函数名、类名等精确锚点。技术互补：在纯自然语言问答中，语义理解仍有优势。

4G 服务器部署方案

最终架构

Nginx → FastAPI → PostgreSQL + pgvector
                   ↓
                  Redis

内存分配建议

服务 内存建议
PostgreSQL512MB
Redis256MB
FastAPI512MB~1GB
Nginx<100MB
Linux 系统1GB
预留缓存1GB

生产环境建议

  • 不暴露 PostgreSQL 和 Redis 端口
  • 使用 HTTPS（Let's Encrypt + Certbot）
  • worker 不要超过 1（uvicorn workers=1）
  • 4G 禁止：Elasticsearch、Kafka、Milvus、ClickHouse、Ollama 大模型

后续扩展（8G+）

Celery/Dramatiq（后台任务）、MinIO（文件存储）、Prometheus + Grafana（监控）、Loki（日志）、Traefik（自动 HTTPS）
